How should we structure the frontend?

All three use React + TypeScript + Tailwind + shadcn/ui with Laerdal brand colours

A

Laerdal Platform — Two Sections Recommended

One app, two areas in the left nav: Implementer (org tree + manifest lifecycle, wired to your real running API) and CPR Finder (polished discovery + booking mock). Shared Laerdal design system, shared header, same brand feel throughout.

Pros

  • Shows full-stack: real API + polished mock
  • Tells a complete Laerdal story in one app
  • Hiring manager sees both backend work AND CPR Finder thinking

Cons

  • Most ambitious — needs good time management
  • CPR Finder section is mock data, not a real API
B

Implementer Admin Only — CPR-Inspired UX

Build only the org/manifest admin UI against your real API, but style it with CPR Finder UX patterns — cards for orgs instead of raw tables, a step-by-step wizard for publishing a manifest, a hierarchy map instead of a flat list. Everything works live. Nothing is mock.

Pros

  • 100% real data, no mocks
  • Safest scope for 1-2 days
  • Deep quality over broad coverage

Cons

  • Doesn't directly show CPR Finder thinking
  • Less obvious connection to the job posting
C

CPR Finder Frontend Only

Abandon the backend connection. Build a full CPR Finder mock — map-based station discovery, booking flow with date/time picker, digital certification screen. No real API, but visually the closest to the actual role.

Pros

  • Most directly relevant to the job posting
  • Impressive user-facing UI/UX showcase

Cons

  • Disconnects from the backend work you already did
  • All mock data — nothing actually runs end-to-end
  • Loses the "full-stack" story
